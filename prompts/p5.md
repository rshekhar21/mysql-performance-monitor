Create a production-grade, one-command deployment system for this application, specifically for an Ubuntu VPS managed through CloudPanel.

The target environment already has a separate production MySQL server running on host port 3306. Do not disturb or replace that MySQL server.

Create:

1. docker-compose.production.yml
2. .env.production.example
3. scripts/install-production.sh
4. scripts/deploy-production.sh
5. scripts/backup-production.sh
6. scripts/restore-production.sh
7. apps/web/nginx.conf with React SPA fallback
8. docs/cloudpanel-deployment.md
9. docs/production-upgrade.md

Requirements:

- Use a separate containerized MySQL database only for the monitoring application's own data.
- Do not publish the app database port to the VPS.
- Remove the test-mysql service from production.
- Bind API, collector health, and web ports only to 127.0.0.1.
- Use configurable host ports with safe defaults:
  - web host port 8085
  - API host port 4400
  - collector health host port 4410
- The internal container ports may remain unchanged.
- Use restart: unless-stopped.
- Add health checks for app-db, API, collector, and web.
- Add depends_on health conditions where supported.
- Build the React frontend with the production domain:
  https://mysqlmonitor.myebs.in
- Include the Nginx SPA fallback:
  try_files $uri $uri/ /index.html;
- Ensure production builds never fall back to localhost API URLs.
- Generate strong JWT_SECRET and CREDENTIAL_ENCRYPTION_KEY during first install.
- Do not overwrite an existing production .env file.
- Do not overwrite or delete an existing Docker volume.
- Back up the application database before migrations or upgrades.
- Run migrations automatically during deployment.
- Seed the first admin only when no admin exists.
- Make scripts idempotent and safe to run repeatedly.
- Use set -euo pipefail.
- Print clear success and failure messages.
- Check that required ports are free before first deployment.
- Detect whether docker compose v2 or docker-compose v1 is available.
- Prefer Compose v2 and document installation.
- Avoid the old Docker Compose ContainerConfig recreation bug.
- Rebuild and recreate only services that changed.
- Never recreate app-db unnecessarily.
- Add commands for:
  - first installation;
  - normal upgrade;
  - viewing logs;
  - stopping;
  - restarting;
  - checking health;
  - backing up;
  - restoring.
- Provide the exact CloudPanel reverse-proxy configuration:
  - / and frontend routes → http://127.0.0.1:8085
  - /api/ → http://127.0.0.1:4400
  - /health and /ready → http://127.0.0.1:4400
- Include the complete tested CloudPanel Vhost configuration.
- Test the scripts on a clean Docker environment.
- Test an upgrade without data loss.
- Test a container restart.
- Test direct navigation to /servers.
- Run lint, typecheck, tests, builds, Docker startup, migrations, health checks, and a smoke test.
- Do not claim completion until the production deployment process succeeds from a clean checkout.

The final intended first deployment should be approximately:

cp .env.production.example .env.production
./scripts/install-production.sh

The intended future update should be approximately:

git pull
./scripts/deploy-production.sh
