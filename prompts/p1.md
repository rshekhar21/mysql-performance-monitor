You are a senior software architect and full-stack engineer. Build a production-oriented MySQL monitoring and performance dashboard.

This must not be a toy, demo, mock dashboard, or single-file prototype. Establish a rock-solid, maintainable foundation suitable for monitoring real MySQL servers and extending into a commercial product.

# 1. Product objective

Create a web application that monitors one or more MySQL database servers and displays:

- server availability and uptime;
- current connection utilization;
- active, sleeping, and maximum connections;
- queries per second;
- reads and writes per second;
- transactions, commits, and rollbacks;
- slow-query activity;
- query latency and query execution summaries;
- running queries and process-list activity;
- InnoDB buffer pool utilization;
- buffer pool reads, hits, dirty pages, and flush activity;
- row reads, inserts, updates, and deletes;
- table locks, metadata locks, deadlocks, and lock waits;
- temporary tables and disk-based temporary tables;
- network bytes received and sent;
- database sizes;
- table sizes;
- index sizes;
- data-versus-index distribution;
- database and table growth over time;
- largest databases and tables;
- storage pressure;
- replication health when replication is available;
- configurable warnings and critical alerts;
- historical trends using charts and graphs.

The system must distinguish between:

1. Live data fetched directly from a monitored MySQL server.
2. Historical snapshots stored in the monitoring application's database.
3. Derived metrics calculated from counters over time.

Do not confuse cumulative MySQL counters with per-second metrics. For counters such as Questions, Bytes_received, Bytes_sent, Com_select, Com_insert, and similar status values, calculate rates using the difference between two snapshots divided by elapsed time.

# 5. Separate databases and connection pools

The system uses two kinds of database connections.

## Monitoring application database

This stores:

- application users;
- roles;
- monitored server definitions;
- encrypted credentials;
- metric snapshots;
- database-size snapshots;
- table-size snapshots;
- query-performance snapshots;
- alert rules;
- alert events;
- audit logs;
- collector runs;
- application settings.

Use a dedicated pool for the monitoring application's own database.

## Monitored MySQL servers

Each monitored server must have an independent, lazily created connection pool.

Requirements:

- bounded pool size;
- connection timeout;
- query timeout;
- idle cleanup;
- server-specific failure isolation;
- retry with capped exponential backoff;
- circuit-breaker-style failure protection;
- no unlimited queues;
- graceful pool shutdown;
- credentials must never be sent to the frontend;
- credentials must never be written to logs;
- credentials must be encrypted at rest.

Never use one monitored server's connection for another server.

# 8. Collector service

The collector must run independently from the web API.

Do not use API request traffic to trigger routine metric collection.

Collector responsibilities:

- find enabled monitored servers;
- obtain a distributed or database-backed collection lock;
- connect to each server independently;
- collect metrics;
- calculate counter deltas;
- persist snapshots;
- evaluate alert rules;
- record collector-run status;
- continue collecting other servers when one server fails;
- expose health and readiness information;
- support graceful shutdown.

Initial collection intervals:

- server health and status counters: every 15 seconds;
- active connections and running-query summary: every 15 seconds;
- InnoDB metrics: every 30 seconds;
- replication metrics: every 30 seconds;
- database sizes: every 5 minutes;
- table sizes: every 15 minutes;
- query digest summaries: every 1 minute;
- daily storage rollup: once per day;
- retention cleanup: once per day.

Make all intervals configurable.

Prevent overlapping collection runs for the same server and metric group.

For each collection run record:

- server;
- metric group;
- started time;
- finished time;
- duration;
- success or failure;
- number of records collected;
- sanitized error category;
- retry count.

# Mandatory API and collector separation

The API service and collector service must be implemented as separate executable Node.js processes.

They may share TypeScript packages, repositories, database models, configuration utilities, and validation schemas, but they must have separate application entry points and independent process lifecycles.

Required entry points should be conceptually similar to:

- apps/api/src/server.ts
- apps/collector/src/worker.ts

The API service must:

- handle HTTP requests;
- authenticate users;
- enforce permissions;
- manage monitored-server configuration;
- serve current and historical monitoring data;
- manage alerts and administrative operations;
- avoid running routine scheduled metric collection inside the API process.

The collector service must:

- run without an HTTP request being received;
- periodically load enabled monitored-server configurations;
- query monitored MySQL servers using read-only monitoring connections;
- calculate rates and derived metrics;
- save snapshots in the monitoring application database;
- evaluate alert rules;
- record collection success and failure;
- continue operating when the React frontend is closed;
- continue collecting when no user is logged in;
- isolate failures so one unavailable server does not stop other collections.

Routine monitoring must never depend on:

- a browser being open;
- a dashboard page being loaded;
- a frontend refresh timer;
- an API endpoint being manually called.

The React dashboard must normally retrieve chart data from historical snapshots stored by the collector.

Direct live queries are permitted only for explicitly live operations such as:

- connection testing;
- capability detection;
- current running queries;
- manual refresh-now diagnostics.

All services may initially run on the same VPS or in the same Docker Compose deployment, but they must remain separate processes so they can later be scaled, restarted, and deployed independently.

- Never run destructive SQL on monitored servers.
- Never rely on the frontend for metric calculations that affect alerts.
- Never treat cumulative counters as current rates.
- Never assume performance_schema or sys schema is available.
- Never allow one unavailable server to crash the collector.
- Never return unlimited time-series results.
- Never store plaintext passwords.
- Never silently ignore collection errors.
- Never fabricate monitoring values.
